>> Embrace Your Browser's Native Power

Since I value convention, W3C standards, and ad-blockers, I use Mozilla Firefox which has the keyboard shortcut CTRL SHIFT E to display the browser's network requests. Your browser's shortcut may be different but all the popular browsers have a network tab in the developer tools, usually opened by pressing F12. When viewing the network monitoring tab right click on the request and select EDIT AND RESEND. This feature is AMAZING and is essentially the same as repeater but you dont have to install anything, and it's free. If your browser doesn't display EDIT AND RESEND when you right click on a request in the network monitor, go to about:config and make sure u set

devtools.netmonitor.features.newEditAndResend to FALSE

This native feature of the web browser is excellent and when used correctly 11 of 10 hackers agree: it greatly reduces your addiction to and reliance on burp repeater.

It's so easy to use. Similar to how burp repeater defines positions in a http request, with your browser's native tooling you can save the request into a file to produce a quick and easy attack surface wherein which distinct vectors may be defined or you can copy the request as a curl command which is just two steps away from being a viable FFUF command.

Get used to using the browser's native network request features and your power level will rise above 9000.

Sniper-to-FFUF

This is the most basic attack type that can be conducted with BURP Intruder. It takes only one payload set, mapped to a keyword defined in an attack vector. This keyword is arbitrary and can be anything you want, but the default is FUZZ. For the Sniper-to-FFUF example we're going to use FFUF to brute force the POST method of a login form by fuzzing the password part of its JSON payload.

For this to work we require an existing email to use with the password we'll be fuzzing in the request. Fortunately, the juice shop is kind enough to provide us with email addresses in the user reviews.

Click on the apple juice and then expand the reviews to find an email address.



We get the email address, admin@juice-sh.op from the review.

Next, open the dev tools on your browser by pressing F12 and go to the network tab, then try to login with any password. I used the email admin@juice-sh.op and the password katanacollector

Find the POST request to login in your network tab, right click it, and copy as curl.


Depending on your environment, the contents in your clipboard may vary but should be something like,

curl 'http://mall.ninja:3000/rest/user/login' -X POST -H 'User-Agent: Mozilla/5.0 (X11; Linux x86_64; rv:102.0) Gecko/20100101 Firefox/102.0' -H 'Accept: application/json, text/plain, */*' -H 'Accept-Language: en-US,en;q=0.5' -H 'Accept-Encoding: gzip, deflate' -H 'Content-Type: application/json' -H 'Origin: http://mall.ninja:3000' -H 'Connection: keep-alive' -H 'Referer: http://mall.ninja:3000/' -H 'Cookie: language=en; welcomebanner_status=dismiss; cookieconsent_status=dismiss' --data-raw '{"email":"admin@juice-sh.op","password":"katanacollector"}'

which is just two steps away from being viable as a command for FFUF. The curl and --data-raw parts of the command just need to be changed; curl to ffuf -u and --data-raw shortened to -d.

Before using FFUF we require information from the webserver to use with FFUF's filter. To brute force a login request it's best to compare against an invalid password response from the server, so we can filter out that noise and view whatever else.

Create and run a shortened version of the curl command obtained from the browser; using only necessary headers:

curl 'http://mall.ninja:3000/rest/user/login' -X POST -H 'Content-Type: application/json' --data-raw '{"email":"admin@juice-sh.op","password":"katanacollector"}'

The server responds with Invalid email or password. which is the string we can use with the FUFF's regex filter -fr to get rid of any bad login responses containing that string.

For this attack we need to define a vector in the request, and identify where the payloads are.

This is done with a keyword. The keywords can be whatever you like but for this example that keyword is going to be (the default) FUZZ. To identify the position in the vector just write the keyword FUZZ instead of katanacollector where the password is supposed to be in the request.

Now that we've identified a vector with a keyword we need to associate the keyword with a payload of inputs for the fuzzer to use. For this we're going to be using the top 100k most used password list daniel miessler got from the NCSC who got it from troy hunt who compiled other people's dumps.

The pathname is long, and in the interest of brevity in the next command, save this list as 100k.txt (or if you already have it, copy and rename it in the current working directory).

To convert the curl command to a FFUF command: change the curl part to ffuf -u and shorten the --data-raw to just -d. Define the wordlist payload with -w, and set the filter to be on the lookout for bad response string by adding -fr with the string "Invalid email or password." from earlier to get the final form for ffuf:

ffuf -u 'http://mall.ninja:3000/rest/user/login' -X POST -H 'Content-Type: application/json' -d '{"email":"admin@juice-sh.op","password":"FUZZ"}' -w 100k.txt:FUZZ -fr "Invalid email or password."

Notice the word FUZZ where the password would be in the JSON payload, and again the label :FUZZ after the wordlist. This tells FFUF that when it sees the word FUZZ in the request, to use the payload labeled FUZZ for the input. We don't have to do this with only one payload set but it's good practice to get into for when working with many.
